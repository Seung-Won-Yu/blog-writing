TypeScript 7.0 발표

내가 먼저 볼 지점: 빌드가 8~12배 빨라져도 eslint·프레임워크·템플릿 검사 도구가 막히면 팀 전체 생산성은 오히려 떨어집니다. 지금은 실제 저장소를 복제해 CI와 편집기 반응을 측정하고, TypeScript 6 호환 정리를 먼저 해두는 편이 안전합니다.

TypeScript 7.0은 도구체인을 Go 기반 네이티브 코드로 이식한 릴리스로, 대형 프로젝트의 전체 빌드에서 보통 8~12배 속도 향상을 제공합니다.

새로운 `tsc`는 기존처럼 npm의 `typescript` 패키지로 설치하며, 편집기 지원은 LSP 기반 언어 서버로 VS Code, Visual Studio, WebStorm 등에서 동작합니다.

`vscode`, `sentry`, `bluesky`, `playwright`, `tldraw` 같은 코드베이스에서 빌드 시간이 크게 줄었고, 일부 환경에서는 총 메모리 사용량도 낮아졌습니다.

TypeScript 7.0은 아직 안정적인 프로그래밍 API가 없어 `typescript-eslint`, Vue, MDX, Astro, Svelte, Angular 템플릿 타입 검사 등 TypeScript를 내장하는 도구는 TypeScript 6.0 병행이 필요합니다.

새 기본값과 제거된 옵션이 있어 TypeScript 6.0 변경 사항을 먼저 적용하는 편이 전환에 유리하며, TypeScript 7.1에서 새 API 제공이 예정되어 있습니다.

Go 기반 네이티브 TypeScript

TypeScript 7.0은 TypeScript 도구체인의 네이티브 포트로, Go로 작성된 새 코드베이스를 사용합니다. 기존 코드베이스의 구조와 로직을 최대한 유지하여 두 컴파일러 간 결과의 일관성과 호환성을 맞추는 방향으로 포팅되었습니다.

새 코드베이스는 네이티브 코드 속도, 공유 메모리 기반 멀티스레딩, 여러 최적화를 통해 전체 빌드에서 보통 8~12배 빠른 결과를 냅니다. 설치는 기존처럼 npm으로 가능합니다.

개발 피드백 루프 단축

TypeScript는 편집기 로딩, 전체 참조 찾기, 자동 완성, 진단, `tsc`, `--watch` 모드 등 개발 과정 여러 단계에 관여합니다. TypeScript 7은 이 과정 전반의 대기 시간을 줄여 피드백 루프를 짧게 만듭니다.

VS Code 코드베이스에서 오류가 있는 파일을 열었을 때 첫 오류가 보이기까지 걸리는 시간은 약 17.5초에서 1.3초 미만으로 줄어 13배 이상 빨라졌습니다. 편집기 지원은 LSP 기반이며, 새 언어 서버는 멀티스레딩을 활용해 동시 요청을 빠르게 처리합니다.

VS Code 사용자는 TypeScript 7 전용 확장을 설치할 수 있고, 향후 몇 주 안에 VS Code 자체에도 TypeScript 7 지원이 포함될 예정입니다. Visual Studio 최신 버전은 워크스페이스에 따라 TypeScript 7을 자동 활성화합니다.
