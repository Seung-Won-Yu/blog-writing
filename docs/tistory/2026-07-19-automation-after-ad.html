첫 실행은 알림이 아니라 기준 저장이다

감지기를 처음 실행할 때는 비교할 과거가 없다. 이때 현재 페이지 전체를 ‘새 변화’라고 알리면 시작하자마자 잘못된 알림이 온다. 그래서 state.json이 없으면 상태를 BASELINE으로 기록하고 알림 없이 끝낸다. 두 번째 실행부터 이전 해시와 현재 해시를 비교한다. 같으면 UNCHANGED, 다르면 CHANGED다. CHANGED일 때만 difflib으로 이전 문장과 현재 문장의 차이를 만들고 알림 함수로 넘긴다. 알림을 이메일·텔레그램·슬랙으로 바꾸더라도 이 판정 부분은 그대로 쓸 수 있다.

state = json.loads(STATE.read_text()) if STATE.exists() else {}
previous_hash = state.get("hash", "")

if not previous_hash:
    status = "BASELINE"
elif previous_hash == current_hash:
    status = "UNCHANGED"
else:
    status = "CHANGED"

STATE.write_text(json.dumps({
    "hash": current_hash,
    "text": current_text
}, ensure_ascii=False, indent=2))

if status == "CHANGED":
    send_notification(previous_text, current_text)
  • BASELINE: 현재 값을 최초 기준으로 저장하고 알림은 보내지 않는다.
  • UNCHANGED: 이전 해시와 같으므로 기록만 남기고 조용히 끝낸다.
  • CHANGED: 달라진 줄을 만든 뒤 이 실행만 알림 대상으로 넘긴다.
  • ERROR: 연결 실패나 대상 누락은 기준을 덮어쓰지 않고 별도 오류로 남긴다.

HTTP 서버가 ETag나 Last-Modified를 잘 제공한다면 조건부 요청을 추가해 전송량을 줄일 수도 있다. MDN은 If-None-Match나 If-Modified-Since 조건이 맞을 때 서버가 304 Not Modified로 본문을 다시 보내지 않을 수 있다고 설명한다. 다만 304는 서버가 보는 리소스 단위의 변화 여부다. 페이지 전체 ETag가 바뀌어도 내가 보는 신청 카드는 같을 수 있으므로, 네트워크 절약용 조건부 요청과 의미 있는 텍스트 비교는 서로 대체 관계가 아니라 함께 쓸 수 있는 두 단계다.

마감에서 신청 가능으로 바꿔 본 실제 결과

로컬 서버를 연 뒤 첫 실행에서는 59d73bf72749로 시작하는 해시가 기준으로 저장됐고 알림은 없었다. 같은 페이지를 다시 읽은 두 번째 실행도 해시가 같아 UNCHANGED와 알림 없음으로 끝났다. 그다음 신청 상태를 마감에서 신청 가능으로, 남은 자리를 0명에서 12명으로, 다음 확인 문구를 접수 마감으로 바꿨다. 세 번째 실행의 해시는 1489a823f9da로 달라졌고 CHANGED와 알림 대상이 출력됐다. 단순히 ‘달라졌다’고만 쓰지 않고 빠진 줄은 붉게, 추가된 줄은 초록색으로 표시했다.

실제 사이트에 붙일 때 필요한 안전장치

로컬 실험은 원하는 만큼 실행해도 되지만 남의 사이트는 다르다. 먼저 이용 약관과 robots.txt, 공개 접근 가능 여부를 확인하고 로그인 우회나 차단 회피는 하지 않는다. 변화를 몇 초 안에 알아야 하는 특별한 이유가 없다면 확인 간격을 30분이나 1시간으로 두는 편이 낫다. 요청마다 알아보기 쉬운 User-Agent를 보내고 timeout과 재시도 상한을 정한다. 페이지 구조가 바뀌어 선택자가 사라지면 ‘품절 해제’가 아니라 오류로 멈춰야 한다. state.json에는 공개된 문구만 저장하고 계정 정보나 개인화된 응답은 남기지 않는다.

  • 감시 대상: 가격·재고·접수 상태처럼 행동을 바꾸는 문구만 고른다.
  • 확인 간격: 기본은 30분 이상, 사이트 정책과 정보 성격에 맞춰 늘린다.
  • 오류 분리: 404·500·연결 실패·선택자 누락을 CHANGED로 기록하지 않는다.
  • 알림 제한: 같은 변경을 반복 전송하지 않고 새 기준 저장 뒤 다시 비교한다.
  • 복구 방법: state.json을 백업하거나 삭제해 언제든 최초 기준 상태로 되돌린다.

정기 실행은 macOS와 Linux의 cron, Windows 작업 스케줄러, GitHub Actions 같은 도구에 연결할 수 있다. 개인 컴퓨터가 꺼져 있어도 돌아가야 한다면 서버나 Actions가 편하지만, 외부 사이트를 주기적으로 읽는 작업은 실행 환경의 고정 IP와 이용 정책을 함께 확인해야 한다. 처음에는 컴퓨터에서 하루 두세 번 수동 실행해 오탐이 없는지 보는 편이 안전하다. 알림 함수도 바로 붙이지 말고 CHANGED 로그가 원하는 순간에만 생기는지 먼저 확인한다.

# 최초 기준 저장
python3 monitor.py https://example.com/apply --target-id watch-target

# 30분마다 실행하는 cron 예시
*/30 * * * * cd /path/to/monitor && /usr/bin/python3 monitor.py https://example.com/apply --target-id watch-target

페이지가 JavaScript로 렌더링되거나 버튼을 눌러야 값이 나타나면 urllib 대신 Playwright로 해당 요소의 textContent를 읽으면 된다. 수십 개 URL, 프록시, 로그인 단계, 이메일과 메신저 알림이 필요해지면 직접 만든 스크립트를 계속 키우기보다 changedetection.io 같은 검증된 도구를 비교하는 편이 유지보수에 유리하다. 작은 스크립트의 역할은 모든 기능을 대신하는 것이 아니라 내 문제에 필요한 변화 기준을 정확히 발견하는 데 있다.

변경 감지 자동화의 핵심은 페이지를 자주 읽는 기술보다 무엇을 변화로 인정할지 좁히는 기준에 있다. 전체 HTML이 아니라 의미 있는 카드의 텍스트만 정리하고, 첫 실행은 기준 저장으로 끝내며, 오류를 변화로 오인하지 않게 분리하면 불필요한 알림을 크게 줄일 수 있다. 실제 사이트에 적용할 때는 이용 약관과 robots 정책, 적절한 확인 간격을 먼저 확인하고 작은 대상 하나부터 시작하는 편이 안전하다.

직접 확인해보려면자주 새로고침하는 공개 페이지 한 곳에서 가격·접수 상태처럼 정말 필요한 문구 하나를 고르고, 30분 이상 간격으로 기준 저장과 변경 없음 흐름부터 시험해 보자.
