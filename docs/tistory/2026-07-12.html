사람이 유지보수할 것처럼 코드를 작성하라

LLM이 코드를 계속 대신 써주더라도, 중복 조건문과 임시 구현을 그대로 병합하면 이후 생성 코드가 그 관행을 프로젝트 규칙처럼 따라갈 수 있습니다.

접근 권한 확인 로직을 라우트 핸들러, 백그라운드 작업, API 엔드포인트, 웹훅 등에 반복해 넣으면 테스트는 통과해도 유지보수 부담은 그대로 남습니다.

모델은 열린 파일, 기존 패턴, 최근 변경사항을 보고 다음 코드를 만들기 때문에, 복사된 조건문 4개는 다섯 번째 복사본을 부르는 신호가 됩니다.

나중에 리팩터링을 요청해도 LLM이 기존 복사본을 모두 제대로 정리한다고 보장하기 어렵고, 나쁜 패턴은 일회성 실수보다 코드베이스의 스타일에 가까워집니다.

중복 조건문, “god” 함수, “나중에 정리” 병합이 쌓이면 프롬프트만으로 되돌리기 어려워지고 결국 사람이 직접 고쳐야 합니다.

LLM이 참고하는 것은 현재 코드베이스임

LLM은 코드를 “진공 상태”에서 작성하지 않고, 사용자의 코드베이스를 읽습니다. 열린 파일, 이미 존재하는 패턴, 최근 변경사항을 참고합니다.

코드베이스에 병합된 지름길은 “여기서는 이렇게 한다”는 학습 신호가 됩니다. 같은 접근 제어 조건이 여러 위치에 반복될 수 있습니다.

이 조건문은 공유 헬퍼로 추출할 수 있지만, LLM이 만든 코드가 동작하고 테스트가 통과한다는 이유로 그대로 병합될 수 있습니다. 같은 규칙을 가진 다섯 번째 엔드포인트를 요청하면, 모델은 처음부터 설계하기보다 저장소 안의 기존 복사본들을 따라갈 가능성이 커집니다.

유지보수를 LLM에 맡긴다는 착각

“나중에 바꿀 때도 LLM이 해줄 것”이라는 생각은 중복과 코드 냄새를 방치하게 만듭니다. 몇 개의 중복 조건문은 치명적이지 않을 수 있지만, 이런 코드 냄새는 계속 쌓입니다. 중복 조건문, “god” 함수, “나중에 정리”하기로 한 병합 등이 있습니다.

나쁜 패턴이 늘어나면 다음 프롬프트의 결과에도 영향을 주고, 나중에 모든 인스턴스를 LLM이 빠짐없이 고칠 것이라고 믿기 어려워집니다. 유지보수를 LLM에 아웃소싱한다고 생각해도, 실제로는 점점 나빠지는 습관을 반복하도록 만드는 상황이 될 수 있습니다. 사람이 유지보수할 것처럼 코드를 작성해야 하며, LLM이 흡수해 되돌려줄 코드 패턴은 좋은 상태로 유지해야 합니다.

오늘의 정처기 문제

운영체제에서 메모리 관리 기법 중, 가상 메모리를 구현하기 위해 물리 메모리를 고정된 크기의 블록으로 나누고, 프로세스의 논리 주소 공간도 같은 크기로 나누어 관리하는 기법은?

  1. 세그먼테이션(Segmentation)
  2. 정답 · 페이징(Paging)
  3. 스와핑(Swapping)
  4. 단편화(Fragmentation)

해설 페이징은 가상 메모리를 구현하기 위한 메모리 관리 기법입니다. 물리 메모리와 논리 메모리 모두 고정된 크기의 블록(페이지와 프레임)으로 나누어 관리하며, 외부 단편화 문제를 해결하는 데 효과적입니다. 세그먼테이션은 가변 크기의 논리적 단위로 메모리를 관리하는 방식입니다.

오늘의 IT · 개발 · 기획 용어

  • 컨테이너 오케스트레이션 (Container Orchestration) 개발
    여러 컨테이너화된 애플리케이션의 배포, 관리, 확장 및 네트워킹을 자동화하는 기술입니다.
  • 백프레셔 (Backpressure) 개발
    데이터 처리 시스템에서 소비자(Consumer)가 생산자(Producer)의 데이터 생성 속도를 제어하여 과부하를 방지하는 메커니즘입니다.
  • 그릿 (Grit) 기획
    장기적인 목표를 향해 열정과 끈기를 가지고 노력하는 역량을 의미하며, 기획 과정에서 문제 해결과 목표 달성에 중요한 요소입니다.
