모집 공고, 품절 상품, 지원사업 신청 페이지는 궁금할 때마다 새로고침하게 만든다. 문제는 열 번 확인해도 아홉 번은 그대로라는 점이다. 이번 실험에서는 별도 유료 서비스나 계정 없이 Python 표준 라이브러리만으로 페이지의 핵심 문구를 저장하고, 다음 실행에서 값이 달라졌을 때만 ‘알림 대상’으로 표시하는 작은 감지기를 만들었다. 로컬 테스트 페이지를 ‘마감·0명’에서 ‘신청 가능·12명’으로 직접 바꿔 기준 저장, 변경 없음, 변경 감지, 대상 영역 누락까지 네 가지 흐름을 확인했다.

업무자동화 실험

실행 기록 · Python 공식 문서와 로컬 실행 실험 · 2026. 7. 14

Python으로 웹페이지 변경 감지기를 직접 만들고 네 가지 상태를 검증한 기록

특정 영역의 텍스트만 저장해 이전 값과 비교하고, 달라진 실행에만 알림 신호를 만드는 구조를 실제 마감·신청 가능 페이지로 검증했다.

왜 새로고침 대신 변경 감지가 필요한가

모집 공고나 재고 페이지를 직접 확인하는 방식은 단순하지만 확인 횟수만큼 집중이 끊긴다. 반대로 페이지 전체를 1분마다 비교하면 날짜·추천 상품·방문자 수처럼 관심 없는 값에도 반응하고 상대 사이트에도 불필요한 요청을 만든다. 필요한 것은 ‘페이지가 조금이라도 달라졌는가’가 아니라 ‘내가 기다리는 신청 상태나 가격이 달라졌는가’를 판단하는 작은 장치다. 변경 감지와 알림 시스템에 관한 조사 연구도 동적인 웹페이지에서 수집, 변화 판정, 알림이 서로 다른 단계라는 점을 구분한다. 이 실험도 읽기와 비교, 알림 대상을 분리했다.

테스트 대상은 주말 사진 클래스 신청 카드다. 처음에는 ‘마감·0명·다음 확인’이 보이고, 나중에는 ‘신청 가능·12명·접수 마감’으로 바뀐다. 페이지의 머리글이나 배경은 비교하지 않고 id가 watch-target인 카드 속 글자만 읽는다. 이 선택 하나가 알림 품질을 좌우한다. 광고 배너나 시계처럼 자주 달라지는 요소가 감시 범위에서 빠지기 때문이다. 실제 사이트에서는 브라우저 개발자 도구로 원하는 영역의 고유 id나 CSS 선택자를 먼저 확인하면 된다.

전체 페이지가 아니라 핵심 문구만 비교한다

비교 범위는 좁을수록 좋지만 너무 좁으면 맥락을 잃는다. 가격 숫자 하나만 보면 통화나 옵션이 바뀌어도 놓칠 수 있고, 카드 전체를 보면 버튼 문구와 재고 수량을 함께 판단할 수 있다. 이번에는 카드 내부의 모든 텍스트를 줄 단위로 모은 뒤 연속 공백과 빈 줄을 제거했다. 같은 문장이 화면 폭 때문에 두 줄로 보이더라도 HTML 속 텍스트가 같으면 같은 값으로 처리한다. JavaScript로 나중에 그려지는 페이지, 로그인 뒤 화면, 무한 스크롤은 urllib만으로 읽히지 않을 수 있으므로 그때는 Playwright 같은 실제 브라우저 방식이 필요하다.

Python 표준 라이브러리로 직접 실험하기

준비물은 Python 3 하나뿐이다. urllib.request는 URL을 요청해 응답을 읽고, html.parser는 지정 영역의 글자를 모으며, hashlib.sha256은 정리된 텍스트를 짧고 고정된 값으로 바꾼다. SHA-256은 여기서 암호를 저장하려는 목적이 아니라 이전 내용과 현재 내용이 같은지 빠르게 비교하는 지문 역할을 한다. 원문도 state.json에 함께 남겨야 변경이 생겼을 때 어떤 줄이 달라졌는지 보여 줄 수 있다. 해시만 저장하면 변화 여부는 알 수 있지만 사람에게 설명할 차이를 만들기 어렵다.

from urllib.request import Request, urlopen
from html.parser import HTMLParser
import hashlib, json, re

def normalize(text):
    return "\n".join(
        line.strip()
        for line in re.sub(r"[ \t]+", " ", text).splitlines()
        if line.strip()
    )

def digest(text):
    return hashlib.sha256(text.encode("utf-8")).hexdigest()

request = Request(URL, headers={"User-Agent": "ChangeMonitorDemo/1.0"})
with urlopen(request, timeout=10) as response:
    markup = response.read().decode("utf-8")

# HTMLParser로 #watch-target의 글자만 모았다고 가정
current_text = normalize(extracted_text)
current_hash = digest(current_text)

Python 공식 urllib 문서는 Request에 헤더를 넣을 수 있고 urlopen 결과를 파일처럼 읽을 수 있다고 설명한다. 실제 감지기에는 timeout을 반드시 둬야 네트워크가 느릴 때 작업 전체가 멈추지 않는다. HTTPError와 URLError도 ‘페이지가 바뀌었다’로 처리하면 안 된다. 서버 오류나 연결 실패는 변화가 아니라 확인 실패다. 이번 스크립트는 감시 대상 영역이 없거나 요청에 실패하면 state.json을 덮어쓰지 않고 종료 코드 2를 반환하도록 만들었다. 다음 정상 실행에서 기존 기준과 비교할 수 있게 하기 위해서다.

첫 실행은 알림이 아니라 기준 저장이다

감지기를 처음 실행할 때는 비교할 과거가 없다. 이때 현재 페이지 전체를 ‘새 변화’라고 알리면 시작하자마자 잘못된 알림이 온다. 그래서 state.json이 없으면 상태를 BASELINE으로 기록하고 알림 없이 끝낸다. 두 번째 실행부터 이전 해시와 현재 해시를 비교한다. 같으면 UNCHANGED, 다르면 CHANGED다. CHANGED일 때만 difflib으로 이전 문장과 현재 문장의 차이를 만들고 알림 함수로 넘긴다. 알림을 이메일·텔레그램·슬랙으로 바꾸더라도 이 판정 부분은 그대로 쓸 수 있다.

state = json.loads(STATE.read_text()) if STATE.exists() else {}
previous_hash = state.get("hash", "")

if not previous_hash:
    status = "BASELINE"
elif previous_hash == current_hash:
    status = "UNCHANGED"
else:
    status = "CHANGED"

STATE.write_text(json.dumps({
    "hash": current_hash,
    "text": current_text
}, ensure_ascii=False, indent=2))

if status == "CHANGED":
    send_notification(previous_text, current_text)
  • BASELINE: 현재 값을 최초 기준으로 저장하고 알림은 보내지 않는다.
  • UNCHANGED: 이전 해시와 같으므로 기록만 남기고 조용히 끝낸다.
  • CHANGED: 달라진 줄을 만든 뒤 이 실행만 알림 대상으로 넘긴다.
  • ERROR: 연결 실패나 대상 누락은 기준을 덮어쓰지 않고 별도 오류로 남긴다.

HTTP 서버가 ETag나 Last-Modified를 잘 제공한다면 조건부 요청을 추가해 전송량을 줄일 수도 있다. MDN은 If-None-Match나 If-Modified-Since 조건이 맞을 때 서버가 304 Not Modified로 본문을 다시 보내지 않을 수 있다고 설명한다. 다만 304는 서버가 보는 리소스 단위의 변화 여부다. 페이지 전체 ETag가 바뀌어도 내가 보는 신청 카드는 같을 수 있으므로, 네트워크 절약용 조건부 요청과 의미 있는 텍스트 비교는 서로 대체 관계가 아니라 함께 쓸 수 있는 두 단계다.

실제 사이트에 붙일 때 필요한 안전장치

로컬 실험은 원하는 만큼 실행해도 되지만 남의 사이트는 다르다. 먼저 이용 약관과 robots.txt, 공개 접근 가능 여부를 확인하고 로그인 우회나 차단 회피는 하지 않는다. 변화를 몇 초 안에 알아야 하는 특별한 이유가 없다면 확인 간격을 30분이나 1시간으로 두는 편이 낫다. 요청마다 알아보기 쉬운 User-Agent를 보내고 timeout과 재시도 상한을 정한다. 페이지 구조가 바뀌어 선택자가 사라지면 ‘품절 해제’가 아니라 오류로 멈춰야 한다. state.json에는 공개된 문구만 저장하고 계정 정보나 개인화된 응답은 남기지 않는다.

  • 감시 대상: 가격·재고·접수 상태처럼 행동을 바꾸는 문구만 고른다.
  • 확인 간격: 기본은 30분 이상, 사이트 정책과 정보 성격에 맞춰 늘린다.
  • 오류 분리: 404·500·연결 실패·선택자 누락을 CHANGED로 기록하지 않는다.
  • 알림 제한: 같은 변경을 반복 전송하지 않고 새 기준 저장 뒤 다시 비교한다.
  • 복구 방법: state.json을 백업하거나 삭제해 언제든 최초 기준 상태로 되돌린다.

정기 실행은 macOS와 Linux의 cron, Windows 작업 스케줄러, GitHub Actions 같은 도구에 연결할 수 있다. 개인 컴퓨터가 꺼져 있어도 돌아가야 한다면 서버나 Actions가 편하지만, 외부 사이트를 주기적으로 읽는 작업은 실행 환경의 고정 IP와 이용 정책을 함께 확인해야 한다. 처음에는 컴퓨터에서 하루 두세 번 수동 실행해 오탐이 없는지 보는 편이 안전하다. 알림 함수도 바로 붙이지 말고 CHANGED 로그가 원하는 순간에만 생기는지 먼저 확인한다.

# 최초 기준 저장
python3 monitor.py https://example.com/apply --target-id watch-target

# 30분마다 실행하는 cron 예시
*/30 * * * * cd /path/to/monitor && /usr/bin/python3 monitor.py https://example.com/apply --target-id watch-target

페이지가 JavaScript로 렌더링되거나 버튼을 눌러야 값이 나타나면 urllib 대신 Playwright로 해당 요소의 textContent를 읽으면 된다. 수십 개 URL, 프록시, 로그인 단계, 이메일과 메신저 알림이 필요해지면 직접 만든 스크립트를 계속 키우기보다 changedetection.io 같은 검증된 도구를 비교하는 편이 유지보수에 유리하다. 작은 스크립트의 역할은 모든 기능을 대신하는 것이 아니라 내 문제에 필요한 변화 기준을 정확히 발견하는 데 있다.

변경 감지 자동화의 핵심은 페이지를 자주 읽는 기술보다 무엇을 변화로 인정할지 좁히는 기준에 있다. 전체 HTML이 아니라 의미 있는 카드의 텍스트만 정리하고, 첫 실행은 기준 저장으로 끝내며, 오류를 변화로 오인하지 않게 분리하면 불필요한 알림을 크게 줄일 수 있다. 실제 사이트에 적용할 때는 이용 약관과 robots 정책, 적절한 확인 간격을 먼저 확인하고 작은 대상 하나부터 시작하는 편이 안전하다.

직접 확인해보려면자주 새로고침하는 공개 페이지 한 곳에서 가격·접수 상태처럼 정말 필요한 문구 하나를 고르고, 30분 이상 간격으로 기준 저장과 변경 없음 흐름부터 시험해 보자.
