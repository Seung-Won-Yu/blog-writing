# 쑥쑥자라나라 데일리 뉴스 편집 계약

이 문서는 매일 실행되는 Codex 편집자의 **유일한 작업 계약**입니다. 예약 프롬프트는 이 문서를 실행하라고만 지시하며, 품질·중복·디자인 규칙을 중복해서 적지 않습니다. Python은 후보 수집과 결정적 검사를 담당하고, Codex는 기사 선정·원문 확인·집필·이미지 기획·검수를 맡습니다. GitHub Actions는 완성 결과의 테스트와 GitHub Pages 배포만 담당합니다.

## 운영 흐름

1. `git pull --ff-only origin main`으로 최신 상태를 받은 뒤 다음 가드를 가장 먼저 실행합니다.

   ```bash
   python3 -m blog_pipeline.publishing.daily_guard --today
   ```

   - `COMPLETE`: 즉시 종료합니다. 원문 재확인, 재집필, 이미지 재생성, 테스트, 커밋, 푸시를 다시 하지 않습니다.
   - `PARTIAL`: 출력된 `reasons`에서 빠진 단계만 복구합니다. 이미 유효한 JSON·이미지·HTML은 다시 만들지 않습니다.
   - `NEW`: 아래 전체 흐름을 한 번만 수행합니다.

2. GitHub Actions가 07:17 KST에 갱신한 `docs/inbox/latest.json`을 읽습니다. 파일의 `day`가 당일 날짜와 다르면 수집기를 한 번만 실행한 뒤 다시 확인합니다. 원뉴스 후보는 최신 작업 입력만 유지하며, 장기 중복 판정은 아래의 가공 완료 글을 기준으로 합니다.
3. `selected` 3건과 필요할 때만 `candidates`의 대체 후보를 확인합니다. 선택한 원문을 직접 열어 제목, 날짜, 핵심 수치, 조건, 한계를 확인합니다.
4. 최근 14일의 `data/days/*.json`에서 제목과 URL만 비교합니다. 같은 canonical URL은 무조건 제외하고, 제목이 거의 같은 동일 사건도 제외합니다. 후속 보도라면 새로 확인된 변화가 제목과 본문에 명확해야 합니다.
5. `data/days/YYYY-MM-DD.json`을 한 번에 작성한 직후, 이미지 생성 전에 중복 가드를 실행합니다.

   ```bash
   python3 -m blog_pipeline.publishing.daily_guard --today --check-duplicates
   ```

   실패하면 중복 기사만 대체하고 다시 검사합니다. 중복 상태로 이미지·HTML을 만들지 않습니다.
6. 기사 이해를 돕는 대표 이미지 1장과 기사별 본문 이미지 3장을 Codex 이미지 생성으로 한 번씩 만들어 당일 assets 폴더에 저장하고 `images` 경로를 기록합니다. 대표 이미지는 첫 기사 하나만 표현하며 세 기사를 한 장에 섞지 않습니다. 각 이미지 생성 전에 다음 짧은 브리프를 `visual.stories`에 기록합니다.

   - `label`: 이미지가 설명할 기사 핵심
   - `scene_label`: 기사 고유 시각 단서 2~4개가 드러나는 한 장면
   - `steps`: 원인 → 결과, 이전 → 이후, 비교, 시간, 데이터 흐름 중 기사 이해에 필요한 관계 하나

   브리프의 실제 물건·기기·문서·행동·제약을 그대로 보여 줍니다. 추상 과정이나 비교는 깔끔한 반실사 편집 일러스트를, 물리적 대상이나 사람의 행동 자체가 뉴스일 때만 증거 중심의 현실 장면을 우선합니다. 노트북 앞 사람, 일반적인 개발자 책상, 맥락 없는 차트·대시보드·서류, AI 로봇·빛나는 뇌·회로 이미지는 금지합니다. 인물은 행동이 기사 핵심일 때만 쓰며 정면 얼굴보다 손이나 어깨 너머 장면을 우선합니다. 영화 같은 조명, 초현실 합성, 네온, 광택 나는 스톡 사진, PPT 카드, 큰 문구 중심 썸네일도 쓰지 않습니다.

   생성 직후 이미지마다 `1초 안에 어느 기사인지 판별되는가`, `브리프의 시각 단서와 관계가 실제로 보이는가`, `다른 기술 기사에도 붙일 수 있는 포괄 이미지가 아닌가`, `대표와 본문 이미지의 구도가 겹치지 않는가`를 확인합니다. 하나라도 실패하면 실패한 이미지만 다시 생성하고, 재생성 프롬프트에는 빠진 대상과 관계를 더 구체적으로 적습니다. Codex 이미지 생성 자체가 실패했을 때만 `generate_editorial_images --today`를 대체 이미지 생성기로 한 번 사용합니다.
7. 원본 이미지를 Git에 올리기 전에 아래 명령으로 `1200×630 WebP`, 장당 최대 256KB, 하루 총 1MB의 `webp-v1` 정책으로 변환한 뒤 HTML과 미리보기를 한 번 만듭니다.

   ```bash
   python3 -m blog_pipeline.publishing.optimize_images --today
   python3 -m blog_pipeline.publishing.export_tistory --today
   python3 -m blog_pipeline.publishing.build_copy_page
   ```

8. 변경 범위 테스트와 미리보기를 확인한 뒤 전체 테스트는 마지막에 한 번만 실행합니다.

   ```bash
   python3 -m unittest discover -s tests
   ```

9. 최종 가드가 `COMPLETE`인지 확인합니다.

   ```bash
   python3 -m blog_pipeline.publishing.daily_guard --today --require-complete
   ```

10. 모든 기준을 통과했고 실제 diff가 있을 때만 하나의 커밋으로 `main`에 한 번 푸시합니다. 완료 후 같은 날짜를 다시 다듬거나 두 번째 커밋을 만들지 않습니다. 티스토리에는 자동 발행하지 않습니다.

## 단일 실행 원칙

- 하루 산출물의 완료 표시는 별도 메모가 아니라 `daily_guard`의 `COMPLETE` 판정입니다.
- `COMPLETE`인 날짜는 읽기 전용입니다. 사용자가 명시적으로 재작성을 요청하지 않는 한 수정하지 않습니다.
- 자동 작업 중 재시도가 발생해도 `PARTIAL`의 누락 단계만 이어서 수행합니다.
- 같은 날짜의 `data/days` 파일을 처음부터 두 번 쓰지 않습니다. 같은 이미지를 정상 상태에서 두 번 생성하지 않습니다.
- 커밋 직전 `git diff`가 없으면 성공으로 종료하며 빈 커밋이나 불필요한 푸시를 하지 않습니다.
- 한 실행에서 커밋 1회, 푸시 1회를 넘기지 않습니다.

## 토큰 절약 원칙

- 자동 작업 프롬프트는 `pull → daily_guard → 이 문서 실행 → 최종 guard → 1회 push`만 지시합니다.
- 기본 입력은 당일 후보 JSON, 선택한 원문 3개, 최근 14일의 제목·URL입니다.
- 저장소 전체, 과거 글 전체, 레거시 생성기, 이미지 바이너리는 문맥으로 읽지 않습니다.
- 기사 3건을 따로 생성하지 않고 하나의 `data/days/YYYY-MM-DD.json`으로 한 번에 작성합니다.
- 결정적 Python 도구가 HTML·광고 분할·미리보기·검사를 담당합니다. Codex는 기사 선정, 글, 이미지 판단에만 토큰을 씁니다.
- 테스트는 변경 범위의 빠른 검사 후 최종 전체 테스트 한 번만 실행합니다. `COMPLETE`이면 테스트도 재실행하지 않습니다.

## 기사 구성

- 첫 기사: 개발자가 아닌 독자도 관심을 가질 생활·사회·일자리·개인정보·AI 이슈
- 둘째 기사: 바로 적용하거나 판단에 활용할 개발 도구·보안·제품 변화
- 셋째 기사: 구조와 원리를 더 깊게 읽을 기술·오픈소스·연구 이야기
- 세 기사를 억지로 묶지 말고, 공통 질문이 실제로 있을 때만 하나의 흐름으로 연결합니다.
- 기사별로 `무슨 일이 있었나 → 독자에게 어떤 변화인가 → 어디까지 확인됐나`가 자연스럽게 이어져야 합니다.
- 기사 요약을 반복하지 말고 구체적인 장면, 숫자, 사용 조건, 반례나 한계를 넣습니다.
- 전체 글은 약 8~12분 분량을 목표로 합니다. 각 기사는 짧은 요약 뒤 `무슨 일이 있었나`, `왜 중요한가`, `직접 확인할 점`이 충분한 근거와 함께 이어져야 합니다.
- 첫 기사에는 일반 독자가 자신의 상황을 떠올릴 구체적인 장면을 넣고, 둘째·셋째 기사에는 실제 설정·코드·제품 사용 시 확인할 지점을 넣습니다.
- 제목·도입·먼저 보는 핵심·기사 본문·마무리에서 같은 문장을 바꿔 쓰며 반복하지 않습니다.

## 문체

- 블로그 이름과 필명은 `쑥쑥자라나라`입니다. 본문에서 필명이나 역할명을 반복해 소개하지 않습니다.
- 차분한 개발자 편집자가 친구에게 설명하듯 씁니다. 보도자료 말투와 보고서 말투를 피합니다.
- `개발자 편집자의 체크포인트`, `승원의 메모`, `오늘의 메모` 같은 별도 견해 상자를 만들지 않습니다.
- 개인적인 해석은 근거 다음 문장에 자연스럽게 녹입니다. 직접 해보지 않은 일을 체험담처럼 쓰지 않습니다.
- `주목할 만하다`, `시사한다`, `단순히 ~가 아니라`, `결국`, `한편`을 습관적으로 반복하지 않습니다.
- 자동화 사용 여부를 본문에 고지하는 문단을 넣지 않습니다. 출처 링크와 사실·의견의 구분으로 신뢰를 확보합니다.
- 첫 문단에서 결론을 전부 요약하지 않습니다. 독자가 다음 문단을 궁금해할 질문이나 장면으로 시작합니다.

## 사실·저작권 기준

- 원문 문장을 길게 복사하지 않습니다. 확인한 사실을 새 문장으로 설명하고 기사마다 원문 링크를 둡니다.
- 수치에는 조사 대상, 기간, 비교 기준을 함께 씁니다. 조건을 찾지 못하면 수치를 단정하지 않습니다.
- 기사와 공식 자료가 다르면 공식 자료를 우선하고 차이를 밝힙니다.
- 추측은 사실처럼 쓰지 않습니다. 확인되지 않은 내용은 빼거나 확인 한계를 명시합니다.
- 기사 원본 이미지는 사용 허가와 라이선스가 명확할 때만 씁니다. 기본값은 새로 만든 편집 이미지입니다.

## 저장 형식

기존 `data/days/*.json`의 `schema_version: 2` 구조를 그대로 따릅니다. 필수 최상위 필드는 다음과 같습니다.

- `date_label`, `weekday`, `visual` (`stories` 3건에 `label`, `scene_label`, `steps` 포함)
- `editorial.headline`, `opening`, `throughline`, `closing`, `action`
- `news` 3건: `title_kr`, `source`, `url`, `published_at`, `audience_lane`, `selection_reason`, `blurb_kr`, `content`
- `quiz`, `terms`, `generation`, `images`

`generation.provider`는 `codex-agent`, `generation.model`은 실제 사용 모델 ID를 기록합니다. 이미지 최적화 명령이 `generation.image_policy`를 `webp-v1`으로 기록합니다. `author_note` 필드는 금지합니다.

## HTML 디자인 계약

- 디자인의 단일 기준은 `design/tistory/skin-layer.css`입니다. 생성기는 의미 구조와 클래스만 출력하고 `style` 속성, 폭, 색, 여백을 출력하지 않습니다.
- 본문은 `.daily-digest-post[data-digest-version="2"]` 하나로 시작하고, 티스토리 스킨이 이미 출력하는 글 제목을 본문에서 반복하지 않습니다.
- 본문 기준 폭은 스킨 토큰 `--sw-content: 720px`입니다. 기사 이미지·구분선·광고·텍스트가 같은 기준선을 사용하며 카드 안에 다시 좌우 패딩을 중첩하지 않습니다.
- 기사별 구조는 `digest-news-card > figure + digest-news-copy`로 유지합니다. 시각 스타일을 바꾸기 위해 새 래퍼나 인라인 스타일을 만들지 않습니다.
- 기사 제목 목록은 `글 순서` 한 번만 출력합니다. `먼저 보는 핵심`, `3분 미리보기`처럼 같은 목록을 반복하지 않습니다.
- AdFit은 `NEWS 01` 섹션이 완전히 끝난 뒤, `NEWS 02` 시작 전에 최상위 형제로 정확히 한 번 둡니다.
- HTML을 기본모드로 왕복해 서식을 재해석하지 않습니다. 초안 페이지의 최종 HTML을 HTML 모드에 한 번 붙여넣습니다.

## 발행 전 체크

- 제목이 검색어만 나열하지 않고 독자의 질문을 담았는가
- 첫 5문장 안에 구체적인 장면과 확인된 변화가 있는가
- 기사마다 원문 링크, 날짜, 핵심 조건, 한계가 있는가
- 같은 내용을 제목·요약·본문·마무리에서 반복하지 않았는가
- 정처기 선택지에 `1.`부터 `4.`까지 번호가 실제 HTML에 보이는가
- 각 이미지가 1초 안에 해당 기사를 떠올리게 하고, 기사 고유 단서와 관계를 실제로 보여 주는가
- 대표 이미지와 본문 이미지가 같은 구도를 반복하지 않는가
- 이미지가 모두 WebP이고 장당 256KB, 하루 총 1MB 이내인가
- `승원의 메모`, 역할명, 자동화 고지문, 가짜 체험이 없는가
- 데스크톱과 모바일 미리보기에서 제목·이미지·본문·정답이 깨지지 않는가
- 본문·이미지·구분선·광고가 같은 720px 기준선에 있고 중첩 패딩이 없는가
- 본문에 `style=` 속성이나 티스토리 제목과 중복되는 제목이 없는가
- 최근 14일과 같은 URL 또는 사실상 같은 제목의 기사가 없는가
- 최종 가드가 `COMPLETE`이고 커밋·푸시가 각각 한 번뿐인가
